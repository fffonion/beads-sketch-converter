# Pindou Convert Offline PWA Design

Date: 2026-04-20
Status: Approved for planning, pending implementation review

## Goal

Make the existing SPA deployable at `https://yooooo.us/pdd/` as an installable PWA that continues to open and function offline after the user has visited it online at least once.

Success criteria:

- The app can be installed from a supported browser as a standalone web app.
- After the first successful online load, opening `https://yooooo.us/pdd/` while offline still boots the app.
- Core local workflows continue to work offline: image import, processing, editing, preview, chart generation, timer, and export.
- Deployment remains compatible with the existing Vite static build and `PINDOU_BASE_PATH`-driven subpath hosting.

Non-goals:

- Supporting `file://` launches from a raw `dist` folder.
- Making third-party sharing destinations or remote canonical metadata work offline.
- Adding server-side behavior.

## Current Context

The project is a Vite + React + TypeScript SPA with browser-local processing. The app already avoids backend dependencies for its main workflows. The build is subpath-aware through `PINDOU_BASE_PATH`, and deployment target is `yooooo.us/pdd/`.

There is currently no PWA manifest, no service worker registration, and no explicit offline caching layer.

## Recommended Approach

Use `vite-plugin-pwa` with Workbox-managed precaching.

Why this approach:

- It fits the existing Vite build pipeline.
- It handles hashed asset precaching reliably across releases.
- It reduces manual cache-versioning work compared with a handwritten service worker.
- It is the lowest-risk way to support both installation and offline boot on a static subpath deployment.

Rejected alternatives:

1. Handwritten service worker
   More maintenance and higher risk of cache invalidation bugs.
2. App-shell-only caching
   Too weak for the stated requirement because dynamic chunks, WASM, and other build assets could still be missing offline.

## Design

### 1. PWA metadata

Add a web app manifest with:

- `name` and `short_name` for the Pindou app identity
- `display: standalone`
- `start_url` set to the deployed subpath entry
- `scope` limited to `/pdd/`
- `theme_color` and `background_color` aligned with the app shell
- installable icons in at least 192x192 and 512x512 sizes

The manifest should be generated through the Vite build so URLs stay consistent with the configured base path.

### 2. Service worker and precache strategy

Register a service worker from the React entrypoint.

The service worker should:

- precache the SPA shell and build assets generated by Vite
- include JS, CSS, HTML, image icons, and the detector WASM asset
- serve the app shell for navigation requests inside the app scope
- activate cleanly on new deploys without leaving stale asset references behind

Primary strategy:

- precache all revisioned build output
- use navigation fallback for in-scope route loads

This app is mostly static after build, so aggressive precaching is appropriate and simpler than selective runtime caching.

### 3. Base path handling

All PWA paths must resolve correctly under `https://yooooo.us/pdd/`.

That means:

- service worker registration path must respect the Vite base path
- manifest URLs and `start_url` must align with `/pdd/`
- offline navigation fallback must stay inside the deployed scope

The existing `normalizeBasePath()` behavior in `vite.config.ts` should remain the source of truth for subpath hosting.

### 4. Asset generation

The app currently has `public/favicon.svg` but not a full installable icon set. Add generated PNG icons derived from the existing brand mark so browser install surfaces have compatible assets.

Required output:

- app icon 192x192
- app icon 512x512
- optionally an Apple touch icon if the browser/platform benefits from it

### 5. Offline runtime behavior

Expected offline behavior after first online visit:

- The landing screen opens normally.
- Existing local file selection and drag/drop continue to work.
- WASM-backed detection still loads from cache and runs locally.
- Existing localStorage-backed preferences and timer state continue to work.
- Export still works because generation is client-side.

Degraded but acceptable behavior:

- canonical/OG metadata remains irrelevant offline
- chart sharing links may not be meaningful offline, but the app itself remains usable

## Components and File Changes

Expected implementation areas:

- `vite.config.ts`
  Add PWA plugin configuration and base-aware manifest/service worker settings.
- `package.json`
  Add the PWA plugin dependency.
- `src/main.tsx`
  Register the generated service worker.
- `public/` or generated asset location
  Add installable icon assets.
- `tests/`
  Add coverage for base-aware PWA configuration and manifest/service worker assumptions.

The core editor and processing logic should not need behavioral changes.

## Data Flow

Online first-load flow:

1. Browser requests the app at `/pdd/`.
2. SPA assets load from the network.
3. Service worker installs and precaches the app shell plus revisioned assets.
4. Later visits can resolve from cache.

Offline revisit flow:

1. Browser requests `/pdd/`.
2. Service worker intercepts navigation.
3. Cached app shell is returned.
4. Cached JS/CSS/WASM assets are loaded.
5. Existing browser-local processing continues without the network.

## Error Handling

If service worker registration fails:

- the app should continue to run as a normal online SPA
- failure should not block rendering

If the user opens the app offline before any successful online visit:

- offline support is not guaranteed
- this is acceptable and does not require custom fallback UX for the first implementation

If a new deployment invalidates older assets:

- Workbox-managed revisioning should refresh the cache set on next online visit
- the app should avoid referencing hard-coded asset filenames

## Testing Strategy

Follow TDD during implementation.

Initial automated coverage should verify:

- the Vite config emits the expected PWA-related settings for subpath deployment
- the manifest fields resolve to `/pdd/`-compatible values
- service worker registration uses the correct base-aware path
- the production build outputs the manifest and service worker assets

Verification after implementation:

- run targeted tests for the new config and registration logic
- run the full relevant test suite
- run a production build
- inspect `dist/` for manifest, service worker, icons, and WASM presence
- manually verify offline boot using a local preview server and browser offline mode

## Risks and Mitigations

Risk: incorrect subpath registration causes the service worker to miss scope.
Mitigation: derive all paths from the Vite base path rather than hard-coding root URLs.

Risk: WASM is omitted from the precache set.
Mitigation: explicitly include `.wasm` in PWA asset handling and verify the built cache manifest.

Risk: installability fails because only SVG favicon exists.
Mitigation: add PNG app icons at required sizes.

Risk: stale caches after deploy.
Mitigation: rely on revisioned build assets and Workbox precache cleanup.

## Implementation Boundaries

Keep this change focused on offline capability and installability. Do not mix in unrelated UI refactors or editor behavior changes. The existing dirty worktree already contains unrelated in-progress frontend edits and should be left untouched.
